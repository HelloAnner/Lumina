- 本仓库当前默认使用中文沟通、中文注释、英文日志。
- 当前实现采用单镜像部署优先原则，默认以本地 JSON 持久化和本地文件存储完成 MVP。
- UI 需要优先贴合 `ui/ui.pen` 的深色极简风格，避免引入多余视觉元素。
- 每次修改后需要同步说明：问题原因、修复思路、实现差异。
- 阅读器交互默认禁止平滑动画；进入书籍、目录跳转、划线跳转都应直接定位到上次或目标阅读位置。
- 阅读器目录栏与右侧划线栏需要支持拖拽调宽，并记忆用户宽度偏好。
- 阅读器左侧目录需要随正文滚动自动跟随，并尽量让当前阅读位置对应目录项保持在目录面板的中线附近。
- 阅读器右侧划线栏默认停留在列表底部；正文滚动时需要自动跟随当前阅读位置，并尽量让对应划线项保持在面板中线附近。
- 阅读器纵向正文中，章节之间需要使用淡横线做弱分隔，保持深色极简风格。
- 新增高亮后，正文原文中的对应片段也要立即呈现高亮效果；不展示“高亮已添加”提示。
- EPUB/正文内容抽取时必须保留块级结构、段落换行，并解码常见 HTML 实体，避免正文被压成一整段。
- HTML/EPUB 内容解析优先使用结构化解析器（如 parse5）与实体解码库，而不是正则剥标签；高亮定位优先保留章节级 locator（如 chapterHref）再做文本兜底。
- EPUB 正文中的插图需要保留在章节内容流中，按原始出现顺序参与纵向阅读排版；图片前后文本顺序不能被打乱。
- EPUB 正文图片优先做最小可用支持：可直接显示、居中约束宽度、保留基础 alt/宽高信息；封面图与正文插图是两条独立链路。
- PDF 与 EPUB 必须保持独立实现路径：解析、阅读器渲染、定位、高亮数据模型不能为了复用而强行耦合，避免一条链路的修改改坏另一条链路。
- 知识库“观点树”没有分组/文件夹概念，只有观点之间的父子嵌套关系；当焦点在某个观点上时，新建默认在其下创建子观点，焦点在根目录时则在根目录创建。
- 知识库观点树必须支持拖拽排序：支持同级上下移动、拖入其他观点作为子观点，以及拖到根目录最下方成为根观点。
- 知识库笔记查看/编辑区域是未来高频核心工作区，默认需要接近 Obsidian 的暗色双栏工作台体验，避免“透明文本叠加预览”这类不可编辑方案。
- 知识库笔记区需要优先保留扩展性：后续会叠加更多可视化与富样式效果，因此结构上要区分编辑面板、实时预览面板和右侧关联面板。
- 知识库笔记编辑默认要求自动保存、实时渲染，并优先保证输入性能；实现上优先使用稳定的文本输入状态与延迟预览渲染，避免 `contenteditable` 全量回流方案。
