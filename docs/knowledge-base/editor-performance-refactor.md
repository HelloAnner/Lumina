# 知识库编辑器性能与体验重构

> 对应模块：知识库 - 笔记编辑器
> 状态：实施中
> 范围：不新增功能，不改变信息架构，专注性能、交互顺滑度与工程边界收敛

---

## 1. 目标

- 保持现有块模型、现有三栏布局、现有侧边批注/对话能力不变
- 把编辑热路径从“整篇文档快照驱动”收敛为“事务级局部更新驱动”
- 把页面级 React 状态从每次按键更新中移出，降低整页重渲染
- 保持当前 API 兼容，在不扩展功能的前提下提升输入、块切换、拖拽、保存反馈的连续性

## 2. 当前主要瓶颈

### 2.1 全量文档重建

当前 `NoteEditor` 在以下路径频繁做双向全量转换：

- `editor.getJSON() -> tipTapDocToBlocks() -> setBlocks()`
- `blocks -> blocksToTipTapDoc() -> editor.commands.setContent()`

这导致：

- 普通输入也会触发完整块数组重建
- 插入、删除、转换块时会重灌整篇文档
- 焦点、选区、目录、slash 菜单等派生逻辑被迫跟着整篇同步

### 2.2 热状态上提

`KnowledgeClient` 当前持有 `blocks` 作为页面级主状态，并把它传给：

- 编辑器
- 右侧聊天面板
- 底部统计栏
- 块选中逻辑

结果是用户输入会带动页面壳子和侧边栏进入更新链路。

### 2.3 保存链路过粗

- 前端对块数组做整包 `PUT /api/viewpoints/:id/blocks`
- 后端直接替换 `articleBlocks`
- 当前存储层每次 mutation 都会重写整个 `data/app/lumina.json`

在输入密集、块数量较多时，这条链路会持续制造主线程和 I/O 压力。

### 2.4 DOM 读写过密

以下交互仍依赖高频 DOM 查询与测量：

- gutter hover 定位
- 目录激活计算
- 块选中/批注态 class 同步
- 拖拽命中判断

这会放大滚动和鼠标移动时的卡顿感。

## 3. 重构原则

- 编辑器内部状态由 TipTap/ProseMirror 维持，父层只接收节流后的快照
- 块级操作优先走 transaction，不再通过 `setContent` 重建全文
- 派生数据按用途拆分：实时局部、节流统计、异步持久化
- 右侧面板按需读取编辑内容，不参与每次按键渲染
- 不改变现有块类型、现有路由和现有用户可见功能

## 4. 实施方案

### 4.1 编辑器内核收敛

- 新增基于当前 selection 的块级事务工具
- 插入、删除、复制、移动、turn into 改成操作当前 ProseMirror 文档
- slash 菜单读取当前块文本时不再全文反序列化
- 仅在以下场景允许外部 `setContent`
  - 切换 viewpoint
  - 远端 blocks 被外部能力替换

### 4.2 页面容器降温

- `KnowledgeClient` 不再保留旧版块编辑逻辑和遗留定时器
- 页面层只保存：
  - 当前 viewpoint 标识
  - 节流后的块快照
  - 保存状态
  - 字数/块数统计
  - 右侧面板必要上下文

### 4.3 保存策略收敛

- `NoteEditor` 内部维护最近一次保存快照和待保存快照
- 输入只更新本地 editor state
- 保存队列以当前 viewpoint 为粒度节流提交
- 外部能力写入新 blocks 时通过单一入口替换编辑器内容并重置保存基线
- 存储层将 `articleBlocks` 从主数据库 JSON 中拆分为按 viewpoint 独立的块文件，降低单次保存的主库写盘体积

### 4.4 交互派生降频

- gutter hover 使用 `requestAnimationFrame` 节流
- 目录定位缓存 heading DOM，滚动时避免重复全量查询
- 块状态 class 同步只在依赖集变化时做增量处理
- 右侧 chat tab 仅在激活时读取最新快照，不随每次按键重新计算整页 props

## 5. 不做的事

- 不新增多选、嵌套块、数据库块等新能力
- 不改变当前视觉风格和布局结构
- 不把本地 JSON 存储整体迁移到新的数据库方案
- 不修改现有对外 API 语义

## 6. 验收标准

- 常规输入不再触发页面级 `blocks` 实时同步
- 插入、删除、复制、转换块不再通过 `setContent` 重建全文
- 切换 chat / annotations tab 不影响编辑时的输入连续性
- 目录、gutter、保存状态仍保持现有功能
- 现有编辑器相关测试保持通过，并补充针对事务级操作的新测试
